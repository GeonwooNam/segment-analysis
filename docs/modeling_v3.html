모델링 v3 — 어떻게 만들었나

Cascade v2(Public 0.44)의 실패 원인을 진단하고, 상위 1~3등 솔루션의 검증된 요소를 접목해 재설계한 파이프라인의 해설.

2026-07-09작성: 건우 + Claude코드: geonu/preprocess_v3.py 외 3개

1. 문제 구조

고객 40만 명 × 6개월(2018.07–12) × 8개 테이블 = 240만 행 × 858 컬럼. 고객마다 등급 A–E 하나를 예측하고, 지표는 macro-F1 — 다섯 클래스의 F1을 동등 가중 평균한다.

클래스train 고객 수의미
A162최상위. 극소수지만 F1의 1/5을 차지
B2440만 중 24명 — 사실상 바늘 찾기
C~21,000D와 경계가 흐림 — 주요 손실 지점
D~58,000
E~320,00080%. 분리 자체는 쉬움

macro-F1의 함정: E를 완벽히 맞혀도 5점 중 1점일 뿐이다. 점수는 소수 클래스(A·B)와 C↔D 경계에서 갈린다.

2. v2는 왜 0.44였나

v2는 직렬 cascade였다 — Stage 1(E 거르기) → Stage 2(AB/C/D) → Stage 3(A/B). 철학은 "탐색 공간을 단계적으로 좁힌다"였고 방향 자체는 옳았지만, 직렬 배선이 치명적이었다.

Stage 1 → Stage 2 (AB recall 16.7%) → Stage 3 └ 실제 A/B 186명 중 ~155명이 여기서 C/D로 확정 — Stage 3에 도착조차 못 함

앞 단계의 오분류가 복구 불가능하게 전파된다. Stage 3 모델이 아무리 좋아도 입력이 없다. 실제로 B에게 넘어온 후보는 1명이었다.

대회 종료 후 확인: 1등 팀도 "A/B 전용 모델 + C 보정"이라는 같은 판단을 했다. 다른 것은 배선 하나 — 그들은 직렬이 아니라 병렬로 묶고 마지막에 덮어썼다.

3. v3 구조 — 2트랙 병렬

TRACK 1 · BASE

C / D / E 3분류

train에서 A/B 186명을 아예 제거하고 학습. 프리미엄 이상치가 사라져 C↔D 경계에만 집중.

CatBoost · class_weight C=2 (1등 방식) · 5-fold 확률 평균

TRACK 2 · VIP

A / B 전용 모델

① AB vs 나머지 이진 → 낮은 threshold로 후보 풀 추출 (recall ≥ 0.9 최소 풀 자동 탐색)

② 풀 안에서 A vs B 정밀 분류 (B 가중 극대화, ID 단위 판정)

▼ 두 트랙은 서로를 모른 채 독립 실행 ▼
BASE 예측지 위에 VIP의 A/B 판정을 덮어쓰기 → 최종 제출

병렬 구조가 해결하는 것:

  1. 오류 전파 차단 — VIP가 A/B 고객을 놓쳐도 그 고객은 BASE의 C/D/E 예측을 받는다. "잘못된 관문에 갇히는" 일이 없다.
  2. 깨끗한 BASE — A/B는 C/D와 분포가 겹치는 고소비 고객이라, 빼고 학습하면 경계가 선명해진다. 데이터 손실은 0.05%.
  3. 독립 개선 — BASE 튜닝과 VIP 튜닝이 서로 간섭하지 않아 실험 사이클이 빠르다.

안전장치: VIP 덮어쓰기는 Stage① 확률 상위 VIP_CAP=60명으로 제한한다. test 10만 명의 기대 A/B는 비율상 ~46명 — 풀 전체를 덮으면 D·E까지 오염되기 때문.

4. 전처리 — 3등 팀의 레시피

  1. 8개 테이블을 월별 병합, 6개월 세로 결합 (240만 × 858)
  2. 단일값(상수) 컬럼 제거 — 준한이 EDA의 상수 목록과 교차 검증용으로 const_cols_v3.csv 저장
  3. int64→int32, float64→float32 다운캐스팅 — 메모리 절반
  4. 문자열 42개 컬럼 Label Encoding (train+test 범주 합집합 기준)
  5. features/train_v3.parquet 저장 — 이후 모든 실험은 여기서 시작

주의했던 것: sentinel 결측(날짜 10101 등)은 3등처럼 일괄 drop하지 않고 유지했다. 트리 모델이 분기로 처리할 수 있고, 우리 데이터의 결측은 "이용 없음"이라는 정보 그 자체이기 때문.

5. 검증 설계 — v1의 교훈

v1이 OOF 0.85를 찍고 Public 0.48이 나온 원인은 데이터 누수였다(스테이지 간 라우팅에 실제 라벨이 섞임). v3의 원칙:

  • StratifiedGroupKFold(group=ID) — 같은 고객의 6개월 행이 fold를 넘나들지 않게
  • 스테이지 간 전달은 전부 OOF 예측 기준 (test와 동일한 정보 흐름)
  • ID 단위 최종 판정 — 6개월 확률 평균(soft 다수결, 3등 방식)
  • 제출마다 OOF와 Public을 페어로 기록, 갭 ±0.02 초과 시 CV 재점검

파생변수를 가른 기준: 표본이 많은 C/D/E는 트리가 원본 컬럼에서 스스로 경계를 학습하지만, 24명뿐인 B는 조합 패턴을 학습할 표본이 없다. 준한이 EDA가 찾아둔 분리 신호(평잔_할부 sep=18.2, 앱 방문 3배, 포인트 소진형)를 VIP 피처로 명시해 학습 부담을 미리 덜어준다.

7. 남은 판단 지점

  • 전 데이터에서 Stage① threshold와 후보 풀 크기 확인 (디버그 샘플과 다르게 나올 지점)
  • VIP_CAP 조정 — 첫 제출의 OOF/Public 갭을 보고
  • 행 단위 실험(고객 집계 vs 월 행) 결과 → 토요일 팀 공유 안건
  • 합성 데이터 확인 결과(금액 끝자리 균등, 금액·건수 모순 31만 행)에 따라 컬럼 간 비율 파생은 OOF 개선 확인 시에만
시간 배분 원칙 (1등의 통찰): B recall에 매몰되지 않는다. macro-F1 기여도는 C↔D 경계 > A 정밀도 > B 순으로 크다. 1등조차 B 확장은 "1명 외 불가능"으로 결론냈다.
