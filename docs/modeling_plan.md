## 1. 1~3등 솔루션에서 취할 것 / 버릴 것

| 요소 | 출처 | 채택 | 근거 |
|---|---|:---:|---|
| **2트랙 병렬 구조** (Base 모델 + VIP 모델 덮어쓰기) | 1등 | ✅ 핵심 | 우리 v2 병목(Stage 2 오류 전파)을 구조적으로 우회 |
| int64→int32, float64→float32 다운캐스팅 | 2·3등 | ✅ | 메모리 절반, 240만 행 처리 가능 |
| 단일값 컬럼 기계적 제거 (858→750) | 3등 | ✅ | 준한이 EDA의 상수 컬럼 145+개와 교차 검증 |
| 전처리 결과 parquet 중간 저장 | 3등 | ✅ | 모든 실험의 시작점 통일, 반복 속도 확보 |
| C class weight 상향 ([2,1,1]) | 1등 | ✅ | C↔D 혼동 12,213건이 우리 C F1 손실 주원인과 동일 진단 |
| 월 단위 행 유지 + ID별 다수결 집계 | 3등 | 🔶 실험 | 준한이 "월별 테이블" 방향과 일치. 고객 단위 집계와 비교 실험 |
| XGB+LGBM+CatBoost 소프트 보팅 | 2·3등 | 🔶 후순위 | 단일 모델 파이프라인 완성 후 마지막 단계에서 |
| feature importance 상위 300개 선택 | 2등 | 🔶 실험 | 준한이 분리도(sep_score) 기반 선택과 비교 |
| SMOTE 오버샘플링 (A 3만/B 3만) | 2등 | ❌ | 1등도 "SMOTE 효과 없음" 결론. 기존 결정(class_weight 유지)과 일치 |
| Null 컬럼 전부 제거 | 3등 | ❌ | 우리 데이터는 sentinel 결측이라 상황 다름. 준한이 EDA 기준으로 판단 |
| 파생변수 0개 | 3등 | ❌ | A/B 분리 특화 피처(할부잔액 등)는 준한이 EDA로 검증된 시그널 → 유지 |

**1등팀 VIP 모델의 핵심 통찰**: B는 모델로 "찾는" 게 아니라, A/B 후보 풀을 좁힌 뒤 그 안에서만 정밀 분류. B 확장은 1명 이상 불가능하다고 결론 → **B recall에 집착하지 말고 A와 C 개선에 시간 배분**하는 것이 macro-F1 최적 전략.

## 2. 전처리 파이프라인 (`preprocess_v3.py`)

```
Step 1. 테이블 병합 (8개 테이블, ID×기준년월 단위 유지 — 240만 행)
Step 2. 단일값(상수) 컬럼 제거          ← 3등 방식 + 준한이 EDA 상수 목록 교차 확인
Step 3. 다운캐스팅: int64→int32, float64→float32
Step 4. sentinel 결측 처리 (기존 방식 유지 — 일괄 drop 하지 않음)
Step 5. Label Encoding (test에만 있는 범주 → encoder에 추가)
Step 6. features/train_v3.parquet, test_v3.parquet 저장
```

- 저장 후 **모든 모델링 스크립트는 이 parquet에서 시작** (원본 재로딩 금지)
- 목표 크기: ~700 컬럼 × 240만 행, float32 기준 메모리 ~7GB

### 행 단위 A/B 테스트 (초기 1회 결정)

| | (가) 고객 1행 (6개월 집계) — 현행 | (나) 월 1행 + ID 다수결 — 3등 방식 |
|---|---|---|
| 행 수 | 40만 | 240만 |
| 트렌드 | mean/std/slope 파생으로 요약 | 원본 월별 값 그대로 |
| 학습량 | 가벼움 | 6배, GPU 필요 |

→ 동일 피처·동일 CV로 (가)/(나) 각 1회 학습해 OOF macro-F1 비교 후 하나로 통일.
(나)가 이기면 준한이의 "월별 테이블" 방향이 검증되는 것.

## 3. 모델 구조 — Cascade v3 (1등식 2트랙 병렬)

기존 v2의 직렬 3단(Stage1 E → Stage2 AB/C/D → Stage3 A/B)을 폐기하고 병렬 2트랙으로 전환:

```
              ┌──────────────────────────────────┐
              │  Track 1: BASE 모델               │
              │  train에서 A/B 제거 → C/D/E 3분류  │
              │  class_weight = C:2, D:1, E:1     │
              │  5-fold OOF 확률 평균              │
              └──────────────┬───────────────────┘
                             │  전 고객 C/D/E 예측
                             ▼
              ┌──────────────────────────────────┐
              │  Track 2: VIP 모델                │
              │  ① AB vs 나머지 이진, 낮은 threshold│
              │     로 AB 후보 풀 추출 (recall 우선)│
              │  ② 후보 풀 안에서 A vs B 분류       │
              │     (class_weight B 극대화)        │
              └──────────────┬───────────────────┘
                             │  A/B 판정 고객만
                             ▼
              BASE 예측 위에 VIP 예측을 덮어쓰기 → 최종 제출
```

**v2 대비 핵심 차이**
1. Stage 2 오류 전파 제거 — BASE는 A/B를 아예 모르고, VIP가 실패해도 BASE 예측(C/D/E)이 남음
2. VIP ①의 threshold는 **precision 희생, recall 확보** 방향 (후보 풀 수백~수천 명 허용, ②에서 걸러냄)
3. VIP 학습 피처는 준한이 EDA A/B 분리 상위 피처 중심 (아래 4절)

**모델**: 1등을 따라 CatBoost 우선 (범주형 처리 유리, GPU 지원). LightGBM은 비교군.

## 4. 피처 전략

### BASE 모델 (C/D/E) — 최소주의 (3등 방향)
- 전처리된 원본 컬럼 + 기존 집계(mean/std) 위주, 파생 최소화
- C↔D 경계 개선용 **Trend 피처(6개월 기울기)만 추가** (improvements.md 3순위)

### VIP 모델 (A/B) — 준한이 EDA 검증 피처 집중 (improvements.md 방향 A)

| 계열 | 피처 | 근거 (A vs B) |
|---|---|---|
| 잔액 | `평잔_할부_3M`(sep=18.2), `평잔_할부_6M`, `잔액_할부_B0M`, `월중평잔_할부` | B가 A의 4배 |
| 채널 | `방문일수_앱_R6M`, `홈페이지_선결제건수_R6M` (개별 컬럼으로) | B가 A의 3배 |
| 포인트 | `포인트_이용포인트_R12M` | B=소진형, A=적립형 |
| 금리 | `고금리_flag = CA이자율_할인전 > 임계값` | B만 20.67%로 튐 |
| 상환 | `상환개월수_결제일_R6M`, `선결제건수_R6M` | B가 빠르고 많음 |

### 피처 선택
- 1차: 준한이 sep_score 기반 (도메인 근거 있음)
- 2차 실험: XGBoost importance 상위 300개 (2등 방식)와 OOF 비교

## 5. 검증 체계

- **고객 단위 StratifiedKFold 5-fold** (월 행 방식 채택 시 StratifiedGroupKFold, group=ID — 3등 방식)
- 모든 단계 **OOF 기준 라우팅** (v1의 누수 재발 방지 — v2에서 수정 완료된 원칙 유지)
- 기록 지표: macro-F1 + **클래스별 F1** + VIP 후보 풀의 AB recall/pool size
- **로컬 OOF vs Public 갭 추적**: `eval_v3_oof.py`의 **5-class ID 단위 macro-F1**과 Public을 `docs/score_log_v3.md`에 페어로 기록. 갭이 ±0.02 이상이면 CV 재점검. (⚠ base 스크립트 출력은 C/D/E 3-class라 비교 대상 아님 — 2026-07-09 첫 제출에서 확인)
- B는 fold당 ~5명 → B F1은 노이즈로 간주하고 fold별 분산 함께 기록
