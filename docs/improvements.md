# 개선 방향 (2026-07-05)

## 현재 상황 요약

| 모델 | OOF Macro-F1 | Public | 비고 |
|---|---|---|---|
| Baseline (LightGBM, 176피처) | 0.4809 | 0.48287 | B F1=0.000 |
| Cascade v1 | 0.8467 (부풀림) | 0.47649 | 데이터 누수 → 실제 성능은 baseline보다 낮음 |
| Cascade v2 (누수 수정) | 0.4557 | 미제출 | B F1=0.000, Stage2 AB recall=16.7% |

**핵심 문제**: A/B 고객 24~186명을 40만 명 중에서 잡아내는 것. 현재 모든 모델이 B F1=0.000.

---

## 문제 진단

### Cascade v2에서 드러난 구조적 한계

```
Stage 1 (Non-E recall ≥ 0.95)  → Non-E 통과 고객 대량 확보
Stage 2 (AB vs C vs D)          → AB recall 16.7% — 실제 186명 중 52명만 예측
Stage 3 (A vs B)                → Stage 2가 52명 넘겨줬는데 B가 1명 → skip
```

Stage 2가 핵심 병목. 원인 분석:

1. **A/B 특화 피처 부족**: 현재 피처가 `이용금액_mean/std` 위주라, C/D와 구분되는 A/B 고유 신호가 약함
2. **학습 데이터 불균형**: Stage 2 훈련셋 ~8만 명 중 AB 186명 → class_weight 한계
3. **피처 엔지니어링 미흡**: 준한이 EDA에서 발견한 A/B 최강 분리 피처들이 현재 피처셋에 미포함

---

## 개선 방향

### 방향 A — 피처 강화 (우선순위 높음)

준한이 EDA에서 발견한 A/B 분리 최강 피처들이 현재 `build_features.py`에 누락되어 있음.

| `평잔_할부_3M` | 5,462 | **21,736** | sep=18.2 — 전체 1위 |
| `평잔_할부_6M` | 8,601 | **24,760** | sep=15.5 |
| `월중평잔_할부` | 4,139 | **17,356** | sep=15.2 |

→ `build_features.py` 5번(잔액정보)에 할부잔액 계열 전체 추가 필요

### 방향 B — 준한이 제안: 월별 테이블 순차 확장

> "월별로 하나의 테이블이라 생각하고 분석 → 5개 테이블로 확장"

#### 현재 방식 vs 제안 방식

| | 현재 | 제안 |
|---|---|---|
| 단위 | 고객 1행 (6개월 집계) | 월 × 고객 행 (패널 구조) |
| 피처 | B0M → mean/std 요약 | 6개월 각 월의 값을 그대로 피처화 |
| 트렌드 | std(변동성)만 포착 | 순서 정보(증가/감소 추세) 보존 |

#### 구체적 방법 (flat wide 방식)

```
고객당 1행, 컬럼 = {feature}_{month_offset}
예: 이용금액_할부_B0M_m0, _m1, _m2, _m3, _m4, _m5
```

- m0 = 201812, m5 = 201807 (최신 → 과거)
- 6개월 × 선택 피처 수 → 컬럼 수 증가하지만 트렌드 직접 표현
- 장점: Trend 피처(기울기) 없이도 트리 모델이 월별 패턴 학습 가능

#### 대안: Trend 피처만 추가 (plans.md P4-6)

```python
# 기울기 = 6개월 선형 추세
slope = np.polyfit(range(6), monthly_values, 1)[0]
```

- 이용금액 증가 중인 D → C 후보 식별에 유용
- 컬럼 수 최소화하면서 트렌드 포착

### 방향 C — Stage 2 구조 개선

현재 Stage 2 병목의 근본 원인: A/B를 C/D와 구분하는 피처가 `class_weight`로만 보상됨.

#### 옵션 C-1: Stage 2 이진화 (AB 여부)
```
현재: AB(0) vs C(1) vs D(2) — 3-class
개선: AB(1) vs 비AB(0) — binary, AB recall 극대화
이후: AB로 분류된 고객만 A/B/C/D/E 재분류 (별도 모델)
```

#### 옵션 C-2: Stage 2 임계값 튜닝
- 현재: `argmax(prob)` → 가장 높은 클래스
- 개선: AB 클래스 확률 > **낮은 임계값** (예: 0.1)이면 AB로 routing
- AB recall 올리되 정밀도 희생 → Stage 3에서 걸러냄

#### 옵션 C-3: 직접 이진 분류 (B vs 나머지)
- Cascade 없이 **B 여부** 이진 분류 모델 별도 학습
- 전체 40만 명에서 B=24명 탐지 → SMOTE 대신 score 기반 threshold 조정

### 방향 D — 피처 선택 명확화 (준한이 방향)

준한이 EDA 스타일: "상수 컬럼 제거 → 분리도 상위 피처 명시 → 파생"

현재 상태: 176개 피처 중 중요도=0인 컬럼 다수 포함

#### 제거 대상 (준한이 EDA 기반)
- 테이블별 상수 컬럼 합계: 회원(10) + 신용(?) + 승인매출(43) + 잔액(13) + 채널(52) + 마케팅(27) = **145개+ 상수**
- 이 중 현재 피처셋에 포함된 것 확인 후 제거

#### 추가 대상 (분리도 상위)
- `잔액_할부` 계열 (sep 10~18)
- `방문일수_앱_R6M` (sep 1.5 — 0-inflated이라 mean 기준)
- `포인트_이용포인트_R12M`
- `CA이자율_할인전` 파생 플래그

## 우선순위 실행 계획

```
1순위 (즉시): 잔액_할부 계열 피처 추가 → build_features_v3.py
              → A/B 분리에 직접적, 현재 완전 누락

2순위: Stage 2 임계값 튜닝 or 이진화 → cascade_v3.py
       → AB recall 16.7% 개선이 전체 성능의 핵심 병목

3순위: 월별 flat wide 피처 or Trend(기울기) 추가 → 준한이 방향
       → 컬럼 수 증가 감수, 트렌드 정보 직접 입력

4순위: 상수 컬럼 정리 + 피처 선택 → 노이즈 제거
```
