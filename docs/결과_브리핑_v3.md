## 2. 0.8361 → 0.5926 은 "점수 폭락"이 아니다

가장 중요한 포인트. **두 숫자는 애초에 같은 것을 재지 않는다.**

`model_v3_base.py`는 학습 전에 A/B를 제거한다(1등 방식). 그래서 이 스크립트가 출력하는
`OOF macro-F1 — ID 단위`는 C/D/E **3개 클래스**의 평균이다.

```
(0.7716 + 0.7667 + 0.9700) / 3 = 0.8361   ← base가 출력하는 값
(0.7716 + 0.7667 + 0.9700) / 5 = 0.5017   ← A=B=0 이라고 가정한 5-class macro (하한선)
```

리더보드는 5개 클래스 macro-F1이다. Public이 0.5926이므로:

```
A, B 두 클래스의 기여 합 = 0.5926 × 5 − 2.5083 = 0.455   (만점 2.000)
```

즉 **C/D/E는 test에서도 OOF와 거의 똑같이 동작했고**, A/B가 만점의 23%밖에 못 가져왔을 뿐이다.
CV가 무너진 게 아니다.

> ⚠️ `데탑_실행_가이드.md` 7절의 *"갭 ±0.02 이상이면 CV 재점검"* 은
> 서로 다른 지표를 비교하는 규칙이라 이 케이스에 적용되지 않는다.
> 이 규칙을 그대로 따르면 멀쩡한 CV를 계속 의심하게 된다.
> 앞으로는 **combine 로직을 OOF에 그대로 적용한 5-class ID 단위 macro-F1**을 기록할 것.

## 3. 왜 A/B가 약한가 — 추론

OOF에서 `prob_ab_stage1` 상위 240명(=test 60명에 해당)을 열어봤다:

| 실제 라벨 | 인원 |
|---|---|
| A | 57 |
| B | 3 |
| **C** | **155** |
| D | 18 |
| E | 7 |

- 상위 240명 중 실제 A/B는 **60명뿐** (정밀도 0.25).
- 오탐의 대부분이 **C**다.
- 덮어쓴 60명의 stage1 확률은 0.987~0.992로 매우 확신에 차 있는데도 4명 중 3명이 틀렸다.

**추론:** Stage①이 VIP(A/B)를 "고액 이용 C 고객"과 구분하지 못한다.
확률값은 높게 나오지만 그 확신이 실제 정확도로 이어지지 않는다 —
즉 모델이 잡아내는 신호가 "VIP성"이 아니라 "고액성"에 가깝다는 뜻일 수 있다.

### VIP_CAP은 이미 최적이다 (가이드의 조언과 다름)

OOF에서 cap을 쓸어봤다. train 400k ID이므로 test cap 60 ≡ train cap 240.

| cap_train | ~cap_test | precision | recall | AB F1 |
|---|---|---|---|---|
| 120 | 30 | 0.317 | 0.204 | 0.248 |
| 186 | 46 | 0.280 | 0.280 | 0.280 |
| **240** | **60 (현재)** | **0.250** | **0.323** | **0.282** |
| 400 | 100 | 0.203 | 0.435 | 0.276 |
| 900 | 225 | 0.123 | 0.597 | 0.204 |

최적은 cap_test ≈ 65에서 AB F1 0.291. 현재 60에서 0.282.
**곡선이 평평해서 cap을 아무리 만져도 macro-F1 기준 ~0.002 수준의 이득밖에 없다.**
가이드는 이걸 "첫 튜닝 포인트"라 했지만, 데이터를 보면 아니다.

### 덤: 죽은 코드 발견

`model_v3_combine.py`는 `prob_ab_stage1` 상위 `VIP_CAP`명만 잘라 쓴다.
따라서 `model_v3_vip.py`의 threshold 탐색(`TARGET_RECALL = 0.90` → 후보 풀 2,824명)과
그 뒤의 `assert`는 **계산만 되고 제출물에는 전혀 반영되지 않는다.**
(threshold 0.88로 뽑은 601명 풀 중 실제로는 상위 60명만 쓰임.)

## 4. 헤드룸이 어디 있나

macro-F1에서 **각 클래스의 가중치는 똑같이 1/5 = 0.2**다. 소수 클래스가 불리하지 않다.

| 클래스 | 현재 F1 | 개선 시 macro 기대 이득 |
|---|---|---|
| E | 0.970 | 없음 |
| C | 0.772 | 크게 올려도 +0.01 수준 |
| D | 0.767 | 〃 |
| A | ~0.45 *(추정)* | 있음 |
| **B** | **~0 *(추정)*** | **최대 +0.13 ~ +0.2** |

B는 train에 24명, test에 ~6명뿐이다. 우리는 3명을 B로 예측했다.
**오탐 없이 3명만 맞춰도 F1_B ≈ 0.67 → macro +0.13.**
다른 모든 개선을 합친 것보다 크다.

## 5. ⚠️ 위 추론을 과신하지 말 것

3~4절의 해석은 **관측된 숫자에서 끌어낸 가설이지, 검증된 사실이 아니다.**
특히 다음을 유의할 것:

- **A ~0.45 / B ~0 는 역산한 추정치다.** 우리는 test 정답을 모른다.
  `0.455`라는 합만 알 뿐, A와 B에 어떻게 쪼개지는지는 확인할 수 없다.
  B가 0이 아니라 0.3이고 A가 0.15일 수도 있다.
- **OOF와 test의 VIP 성능이 어긋난다.** OOF AB-detection 정밀도는 0.25인데,
  test에서 A는 그보다 훨씬 잘 맞은 것으로 보인다. 즉 **VIP 쪽 OOF 추정은 신뢰구간이 넓다.**
  A/B가 train 400k명 중 186명(0.047%)뿐이라 fold별 분산이 크다.
- **B는 test에 6명 남짓이다.** 한 명 차이로 F1_B가 0.0 ↔ 0.4를 오간다.
  B를 겨냥한 튜닝은 리더보드 점수를 크게 흔들지만, 그게 **실력인지 운인지 구분되지 않는다.**
  Public에 과적합하기 가장 쉬운 지점이다.
- **cap 스윕은 OOF 기준이다.** test에서도 60이 최적이라는 보장은 없다.

> 요약하면: "B에 헤드룸이 있다"는 **산술적으로는 맞지만**,
> 그 헤드룸을 실제로 회수할 수 있는지는 별개 문제다.
> 이 문서를 근거로 B 하나에 올인하지 말 것. 제출 횟수를 태워가며
> B만 쫓다 보면 Public 노이즈를 학습하게 된다.

## 6. 다음에 볼 것 (우선순위 순)

1. **5-class OOF 평가기를 먼저 만든다.** combine 로직을 OOF에 그대로 적용해
   A~E 전부 포함한 ID 단위 macro-F1을 뽑는다. 이게 없으면 앞으로의 모든 튜닝이
   Public 점수에만 의존하게 된다 — 제출 횟수도 아깝고 과적합도 부른다. **가장 먼저 할 일.**
2. **Stage① 피처를 의심한다.** VIP와 고액 C를 가르는 신호(잔액/한도 대비 이용률,
   카드론·리볼빙 패턴 등)가 들어가 있는지 확인. 지금은 "고액성"만 보고 있을 가능성.
3. VIP_CAP은 **건드리지 않는다.** 이득 ~0.002.
4. C/D 혼동 개선은 안전하지만 상한이 낮다 (+0.01 내외).
